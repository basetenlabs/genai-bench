# Feature Comparison: Origin vs Upstream

**Date:** 2026-04-10
**Fork:** `basetenlabs/genai-bench` (origin)
**Upstream:** `sgl-project/genai-bench` (upstream)

Both sides independently implemented solutions for these four problems. This document details how each approach works, what the tradeoffs are, and what to do at merge time.

---

## 1. Prefix Caching Benchmarking

**Problem:** Benchmark KV cache / prefix caching performance by sending requests that share a common prefix.

### Origin: `prefix-repetition` traffic scenario

**Commits:** `0941d33`, `7a8cbf1`
**Key files:** `genai_bench/scenarios/text.py`, `genai_bench/sampling/text.py`

**User-facing interface:** A new scenario type `P(prefix_len,suffix_len)/output_len` (e.g. `P(2000,500)/200`) passed via `--traffic-scenario`. No new CLI flags; it slots into the existing scenario system.

**How it works:**
- `PrefixRepetitionScenario` (in `scenarios/text.py`) is a `Scenario` subclass with `scenario_type = TextDistribution.PREFIX_REPETITION`. Its `sample()` returns a fixed `(prefix_len, suffix_len, output_len)` triple — no randomness.
- In `sampling/text.py`, `_sample_chat_request` detects `isinstance(scenario, PrefixRepetitionScenario)` and delegates to `_sample_prefix_repetition_request`.
- That method uses `_shared_prefix_cache: Dict[str, str]` keyed by `f"prefix_{prefix_len}"`. The prefix is generated once and reused for every request. A per-request unique suffix is generated and combined as `f"{prefix}{separator}{suffix}"` where the separator is `f"\n\n--- Request #{counter} ---\n\n"`.
- Output length is controlled via `min_tokens` and `max_tokens` both set to `output_len` (plus `ignore_eos=True`).

### Upstream: `--prefix-len` / `--prefix-ratio` CLI options

**Commit:** `ed72aac`
**Key files:** `genai_bench/cli/cli.py`, `genai_bench/cli/option_groups.py`, `genai_bench/cli/validation.py`, `genai_bench/sampling/text.py`

**User-facing interface:** Two new flags in `sampling_options()`: `--prefix-len` (absolute token count) and `--prefix-ratio` (fraction of input). Mutually exclusive. Works orthogonally with any existing scenario type (D, N, U).

**How it works:**
- `TextSampler.__init__` accepts `prefix_len` and `prefix_ratio` parameters and stores `self._shared_prefix: Optional[str] = None`.
- `_sample_text` takes an `effective_prefix_len` argument. If `--prefix-len` is set, it generates the prefix once using a seeded `random.Random(hash((42, effective_prefix_len)))` and caches it in `self._shared_prefix`. If `--prefix-ratio` is set, it generates a fresh prefix per request with the same seeded RNG (no caching).
- The separator is a random 4-byte hex string (`random.randbytes(2).hex()`) rather than a human-readable counter string.
- `validate_prefix_options` in `validation.py` enforces: mutual exclusivity, `text-to-text` only, no dataset mode, `--prefix-ratio` requires all-deterministic scenarios, `--prefix-len` validates against minimum possible input tokens per scenario.

### Key Differences

| | Origin (`prefix-repetition`) | Upstream (`--prefix-len`) |
|---|---|---|
| Interface | New scenario type `P(...)` in `--traffic-scenario` | CLI flags `--prefix-len` / `--prefix-ratio` |
| Output length control | Encoded in scenario: `P(prefix,suffix)/output` | From the base scenario (`D(in,out)` etc.) |
| Separator | Human-readable `"\n\n--- Request #N ---\n\n"` | Random hex `randbytes(2).hex()` |
| Prefix generation | Ad-hoc `random` state (non-deterministic across workers) | Seeded `random.Random(hash((42, prefix_len)))` — consistent across distributed workers |
| Validation | None beyond regex pattern matching | Rigorous: checks `prefix_len <= min_input_tokens`, resamples if stochastic scenario falls below |
| Composability | Standalone scenario — can't combine with D/N/U distributions | Orthogonal modifier on top of any scenario type |

### Assessment

They are **redundant in intent** but not directly mergeable. Upstream's design is more principled: composable with existing scenarios, deterministic across distributed workers, with rigorous validation. Origin's `P(...)` scenario is simpler but creates a whole new scenario type that bypasses the existing composition machinery.

**Merge recommendation:** Adopt upstream's `--prefix-len`/`--prefix-ratio`. Consider whether `P(...)` syntax adds enough ergonomic value to keep as syntactic sugar, or drop it.

## 2. Reasoning Token Support

**Problem:** Track and report reasoning/thinking tokens generated by models that support chain-of-thought.

### Origin: GPT-OSS reasoning tag parsing

**Commits:** `cd4085c`, `3b30426`, `99631d3`
**Key file:** `genai_bench/user/openai_user.py`

**How it works:**
In `parse_streaming_response`, the delta content is extracted with a fallback chain:
```python
content = (
    delta.get("content")
    or delta.get("reasoning_content")
    or delta.get("reasoning")
)
```
This treats reasoning content as regular output — it's appended to `generated_text`, not tracked separately. No separate `reasoning_text` buffer. No count. `_get_usage_info` returns only `(effective_prefill, num_prompt_tokens, tokens_received)` — no `reasoning_tokens` extracted from `completion_tokens_details`.

There are no `reasoning_tokens` fields anywhere in `protocol.py` or `metrics.py`.

### Upstream: Full reasoning token reporting across all backends

**Core commit:** `1cc360d` (protocol + metrics), plus per-backend commits (`fd4ec84` OpenAI, `e0f637a` Azure, `aa8e3d8` Cohere, `0cd8b46` Together, `bcceb31` OCI, `81ccb82` Bedrock, `7af3973` Vertex)
**Key files:** `genai_bench/protocol.py`, `genai_bench/metrics/metrics.py`, `genai_bench/metrics/request_metrics_collector.py`, all user files

**How it works:**
- Introduces `BACKEND_REASONING_FIELD` dict: `{"vllm": "reasoning", "sglang": "reasoning_content"}`. Per chunk, checks `delta.get(reasoning_field)` separately from `delta.get("content")`.
- Both are combined into `content` for TTFT/text purposes, but reasoning chunks are additionally accumulated into a separate `reasoning_text` buffer.
- `_get_usage_info` extracts `completion_tokens_details.reasoning_tokens` from usage data and returns a 4-tuple including `reasoning_tokens`. Falls back to tokenizing `reasoning_text` via the sampler with a warning (`"reasoning_tokens_estimated"`).
- `protocol.py`: `UserChatResponse` has `reasoning_tokens: Optional[int] = Field(default=0, ...)`.
- `metrics.py`: `RequestMetrics` has `num_reasoning_tokens: Optional[int]` and `BenchmarkMetrics` has `num_reasoning_tokens: StatField`. Included in `THROUGHPUT_METRICS`.
- `request_metrics_collector.py`: `self.metrics.num_reasoning_tokens = response.reasoning_tokens or 0`.

### Key Differences

| | Origin | Upstream |
|---|---|---|
| Scope | GPT-OSS models only, `openai_user.py` only | All backends (OpenAI, Azure, Cohere, Together, OCI, Bedrock, Vertex) |
| Parsing | Blind fallback chain (`content or reasoning_content or reasoning`) | Backend-aware field lookup via `BACKEND_REASONING_FIELD` dict |
| Tracking | Reasoning content collapsed into `generated_text` | Separate `reasoning_text` buffer alongside `generated_text` |
| Counting | None — no reasoning token count anywhere | Full pipeline: `completion_tokens_details.reasoning_tokens` with tokenizer fallback |
| Reporting | None | `num_reasoning_tokens` in `UserChatResponse`, `RequestMetrics`, `BenchmarkMetrics` |

### Assessment

Upstream's approach **completely subsumes** origin's. Origin's fallback chain reads the same delta fields but doesn't track or report reasoning tokens — it just avoids dropping reasoning content from the generated text.

**Merge recommendation:** Adopt upstream's full reasoning token pipeline. Origin's or-chain becomes redundant once upstream's backend-scoped field lookup is in place. Verify that GPT-OSS model responses match one of the `BACKEND_REASONING_FIELD` keys, and add a new key if needed.

## 3. Metrics Filtering Behavior

**Problem:** Requests with implausibly fast timing values (e.g., `output_inference_speed > 1000 tokens/s`) distort aggregate metrics.

### Ancestor behavior (dacc5be)

`_should_filter_metrics()` was a static method called at the top of `add_single_request_metrics()`. If `output_inference_speed > 1000` tokens/s, it returned `True` and the **entire request was dropped** — not stored in `all_request_metrics`, not counted in `num_completed_requests`, not counted as an error. The request simply vanished from all reporting.

### Origin: Disable filtering entirely

**Commit:** `8f37ff3`
**Key file:** `genai_bench/metrics/aggregated_metrics_collector.py`

The method `_should_filter_metrics()` was gutted — its body was commented out and replaced with `return False`, meaning every request passes through. All requests, even those with `inference_speed > 1000`, are stored and aggregated.

Additional origin changes in this file:
- `OPTIONAL_METRICS_FIELDS` handling: optional network timing fields are processed in a separate `optional_keys` loop that tolerates all-None values, while required `filtered_keys` raises `ValueError` if no values are found.
- `dataset_character_to_token_ratio` parameter added to `aggregate_metrics_data()`, computes `mean_total_chars_per_hour`.
- Guards against `run_duration <= 0` before dividing (prevents `ZeroDivisionError`).

### Upstream: Nullify bad fields instead of dropping requests

**Commits:** `39abdcb`, `9991107`
**Key file:** `genai_bench/metrics/aggregated_metrics_collector.py`

Renamed `_should_filter_metrics()` to `filter_metrics()` (public). Instead of returning `True` to skip the whole request, it **nullifies** the bad fields in-place:
```python
metrics.tpot = None
metrics.output_inference_speed = None
```
The request is still stored and counted; only the two bad fields are zeroed out.

The trigger condition also changed: instead of `inference_speed > 1000`, the filter fires when `output_latency < 0.001 AND tpot != 0` — a latency-based heuristic that avoids false positives on non-streaming tasks (embeddings, image gen) where `tpot=0` is intentional.

Additional upstream changes:
- `import time` / refresh interval: `self._last_update_time` and `self._update_interval` throttle `_update_live_metrics()` calls via `set_run_metadata(metrics_refresh_interval=...)`.

### Practical Impact

| Scenario | Ancestor | Origin | Upstream |
|---|---|---|---|
| Request with implausibly fast latency | Entire request dropped silently | Kept, all fields included | Kept; only `tpot`/`output_inference_speed` set to None |
| Error counts | Affected (request never stored) | Unaffected | Unaffected |
| Token counts / throughput | Skewed low (missing requests) | Accurate | Accurate |
| TPOT / inference speed aggregates | Jitter removed but biased | Potentially inflated by outliers | Outlier values excluded per-field |

### Assessment

**Opposite approaches to the same problem.** Origin said "stop filtering" — simple, but means a request with `inference_speed = 50000` inflates mean TPOT and inference speed stats. Upstream said "filter smarter" — nullify only the affected fields so the request still contributes to all other stats, and protect non-streaming tasks with the `tpot != 0` guard.

**Merge recommendation:** Upstream's per-field nullification is the more correct approach. However, origin's `run_duration <= 0` guard and `OPTIONAL_METRICS_FIELDS` split address real edge cases that upstream doesn't handle — these should be layered on top of upstream's filtering logic.

## 4. Token Count / Prefill Accuracy

**Problem:** The number of tokens reported as "input tokens" doesn't match what the model server actually processes, due to tokenizer inconsistencies and chat template overhead.

### Ancestor behavior (dacc5be)

`_sample_text` concatenated lines with `prompt += line` (no space prefix). When truncating, it decoded `line_tokens[:left_tokens_to_sample]` and appended with a space. It never verified the actual token count after decode — it assumed truncation was exact. `num_prefill_tokens` was computed via `self.get_token_length(prompt)` after sampling and reported as-is.

### Origin: Patch `_sample_text` for edge cases

**Commits:** `fcbaa64`, `ebb6905`, `7a4b445`, `51d036f`, `e81fdae`
**Key file:** `genai_bench/sampling/text.py`

Three specific fixes:

1. **Space-prefix fix during concatenation:** Changed `prompt += line` to `prompt += (" " if prompt else "") + line` and applied the same logic when computing `line_with_space` before encoding. This ensures tokenization during budget accounting matches how lines are actually joined — fixing off-by-one errors where boundary tokens differ when words abut without spaces.

2. **Post-truncation token count verification:** After decoding a truncated slice (`line_tokens[:left_tokens_to_sample]`), origin re-encodes the assembled `prompt` and if `actual_tokens != num_input_tokens`, re-tokenizes the full prompt and truncates again to exactly `num_input_tokens`. This guards against BPE boundary artifacts and whitespace normalization inconsistencies in decode/re-encode round trips.

3. **`min_tokens`/`max_tokens` in `_sample_chat_request`:** Origin adds `no_min_tokens` support, setting `min_tokens` and `max_tokens` from scenario output length.

### Upstream: Structural refactor with chat template awareness

**Commit:** `ee65f3f`
**Key files:** `genai_bench/sampling/text.py`, `genai_bench/user/openai_user.py`

Structural changes rather than accuracy patches:

1. **Extracted `_generate_text_from_dataset`:** The inline loop from `_sample_text` was moved to a dedicated method accepting an optional seeded `random.Random` for deterministic prefix generation across distributed workers.

2. **`_sample_text` orchestrates prefix+suffix assembly:** Handles `effective_prefix_len` logic — generating a cached shared prefix, generating a random hex separator, accounting for separator token length, and concatenating.

3. **Added `_get_prefill_token_count`:** A new method that applies `tokenizer.apply_chat_template(messages, add_generation_prompt=True, tokenize=True)` to count tokens the way the model server actually sees them (i.e., after chat template wrapping including system message tokens, BOS/EOS tokens, role markers). Falls back to `get_token_length(prompt, add_special_tokens=True)` if the tokenizer has no chat template. **Note:** The method is defined but `_sample_chat_request` still calls `self.get_token_length(prompt)` — not yet fully wired in.

4. **Cleaner usage info in `openai_user.py`:** `_get_usage_info` drops the hardcoded `>= 50` token-difference warning in favor of simply preferring `num_prompt_tokens` from the server when non-None.

### Key Differences

| | Origin | Upstream |
|---|---|---|
| Approach | Patch existing code for edge cases | Structural refactor + new counting method |
| Space-prefix fix | Yes — `(" " if prompt else "") + line` | **Partially** — applied in truncation path but NOT in whole-line concatenation (`text += line` without space) |
| Post-truncation re-verify | Yes — re-encode and re-truncate if mismatch | No — not carried over |
| Chat template counting | No | Yes — `_get_prefill_token_count` using `apply_chat_template` (defined but not yet wired in) |
| `min_tokens`/`max_tokens` | Yes, with `no_min_tokens` flag | No |

### Assessment

These are **complementary, not conflicting.** Origin fixed real edge-case bugs in the existing counting path. Upstream added a fundamentally better counting approach (chat template) but didn't port origin's edge-case fixes.

**Merge recommendation:** Adopt both. Upstream's `_get_prefill_token_count` addresses the root cause (raw token count vs. chat-template count) more fundamentally. Origin's space-prefix fix and post-truncation re-verify guard against tokenizer consistency bugs that upstream still has — they should be layered on top. Specifically:
- Port origin's space-prefix fix to upstream's `_generate_text_from_dataset` whole-line concatenation path
- Port origin's post-truncation re-verify into upstream's refactored code
- Wire in upstream's `_get_prefill_token_count` to replace raw `get_token_length` calls
- Keep origin's `min_tokens`/`max_tokens` + `no_min_tokens` support

## 5. Non-Overlapping Features

Features that exist on only one side. No conflict — these merge cleanly.

### Origin-only

| Feature | Key files | Description |
|---------|-----------|-------------|
| **Baseten provider** | `auth/baseten/`, `user/baseten_user.py` | Full auth provider + API client for Baseten's non-OpenAI endpoint. Includes `--disable-streaming`, `BASETEN_API_KEY` env var, API key filtering from metadata. |
| **Async runner** | `async_runner/` (entire package) | Alternative execution engine (`--execution-engine async`) with open-loop and closed-loop modes. Replaces Locust for async workloads. |
| **Multi-model benchmarking** | `cli.py`, `async_runner/base.py` | `--api-model-name` accepts multiple values; outer loop iterates models per run. `--tokenizer-from-model` auto-loads tokenizer. |
| **Network latency metrics** | `async_runner/base.py`, `metrics/metrics.py`, `protocol.py` | Opt-in `--track-network-timing` captures TCP/TLS/DNS timing via aiohttp `TraceConfig`. |
| **Prompt instruction tuning** | `sampling/text.py` | Refined system prompt appended to requests to maximize output token generation. |
| **TRT-LLM engine** | `cli/option_groups.py` | Added `trt-llm` to `--server-engine` choices. |
| **H100 MIG GPU** | `cli/option_groups.py` | Added H100 MIG to allowed GPU list. |
| **Metrics math safety** | `metrics/aggregated_metrics_collector.py`, `metrics/request_metrics_collector.py` | Division-by-zero guards, negative TTFT clamping, zero `run_duration` guard. |
| **CPU-only PyTorch docs** | `docs/getting-started/installation.md` | Reduces torch download from 2GB+ to ~200MB. |
| **Performance troubleshooting guide** | `docs/user-guide/performance-troubleshooting.md` | Benchmarking pitfalls, speculative decoding, dataset guidance. |
